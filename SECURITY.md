# Политика безопасности

## Поддерживаемые версии

| Версия | Поддержка |
|---|---|
| 0.1.x (последняя) | Да |

Мы предоставляем обновления безопасности только для последнего минорного релиза. Настоятельно рекомендуем всегда использовать последнюю версию.

## Сообщение об уязвимости

**Пожалуйста, не создавайте публичный GitHub issue для уязвимостей безопасности.**

### Как сообщить

Отправьте письмо на **security@outpost.dev** со следующей информацией:

1. **Описание** уязвимости
2. **Шаги воспроизведения** (или proof-of-concept)
3. **Оценка воздействия** -- что может сделать атакующий?
4. **Затронутый компонент** (core, gateway, proxy, client, frontend, API)
5. **Предлагаемое исправление** (если есть)

### Что ожидать

| Сроки | Действие |
|---|---|
| **24 часа** | Мы подтверждаем получение вашего сообщения |
| **72 часа** | Мы даём предварительную оценку и рейтинг серьёзности |
| **7 дней** | Мы стремимся подготовить исправление для критических/высоких уязвимостей |
| **14 дней** | Мы координируем раскрытие и выпускаем исправленную версию |
| **30 дней** | Публичное раскрытие (или раньше, если исправление выпущено) |

### Классификация серьёзности

Мы используем систему оценки CVSS v3.1 для классификации уязвимостей:

- **Критическая (9.0-10.0):** Удалённое выполнение кода, обход аутентификации, извлечение данных всех пользователей
- **Высокая (7.0-8.9):** Повышение привилегий, IDOR с доступом к данным других пользователей, утечка ключей WireGuard
- **Средняя (4.0-6.9):** Раскрытие информации, CSRF, отказ в обслуживании
- **Низкая (0.1-3.9):** Незначительные утечки информации, проблемы только на уровне UI

### Что квалифицируется

- Обход аутентификации или авторизации
- Удалённое выполнение кода
- SQL-инъекции, инъекции команд
- IDOR (Insecure Direct Object References) на чувствительных ресурсах
- Утечка приватных ключей WireGuard
- Перехват токенов/сессий
- Криптографические слабости
- Повышение привилегий
- Раскрытие чувствительных данных в логах или API-ответах

### Что не квалифицируется

- Уязвимости в зависимостях без продемонстрированного пути эксплуатации в Outpost
- Отказ в обслуживании через исчерпание ресурсов (если не тривиально эксплуатируемо)
- Проблемы, требующие физического доступа к серверу
- Атаки социальной инженерии
- Отсутствующие заголовки безопасности, не ведущие к продемонстрированному эксплойту
- Уязвимости в сторонних сервисах (PostgreSQL, Redis, модуль ядра WireGuard)

## Bug Bounty

В настоящее время у нас нет формальной программы bug bounty. Тем не менее, мы глубоко благодарны за ответственные раскрытия и готовы:

- Указать вас в security advisory и CHANGELOG (если вы не предпочитаете анонимность)
- Отправить вам мерч Outpost (когда будет доступен)
- Включить вас в наш Зал славы безопасности

Мы рассматриваем возможность запуска формальной программы bounty и обновим этот документ, если она будет запущена.

## Лучшие практики безопасности для операторов

При развёртывании Outpost в продакшене:

1. **Смените учётные данные по умолчанию** сразу после первого входа
2. **Включите MFA** для всех администраторских аккаунтов (TOTP или WebAuthn)
3. **Используйте TLS** для всех HTTP и gRPC эндпоинтов
4. **Ограничьте доступ к базе данных** -- никогда не выставляйте PostgreSQL в публичный интернет
5. **Установите надёжный `JWT_SECRET`** -- минимум 32 байта случайных данных
6. **Включите аудиторское логирование** и перенаправляйте в вашу SIEM
7. **Запустите панель соответствия** для проверки типичных ошибок конфигурации
8. **Поддерживайте Outpost в актуальном состоянии** -- используйте последнюю версию
9. **Используйте сегментацию сети** -- разворачивайте `outpost-proxy` в DMZ
10. **Проверьте ZTNA-политики** -- настройте проверки состояния устройств, подходящие для вашей среды

## Политика раскрытия

Мы следуем процессу координированного раскрытия:

1. Автор сообщения отправляет информацию об уязвимости приватно
2. Мы подтверждаем и оцениваем серьёзность
3. Мы разрабатываем и тестируем исправление
4. Мы выпускаем исправление и публикуем security advisory
5. Автор сообщения получает упоминание (при желании)

Мы просим авторов сообщений предоставить нам разумный срок (до 90 дней) для устранения уязвимостей перед публичным раскрытием.
