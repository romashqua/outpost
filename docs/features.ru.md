## Аутентификация и авторизация

**Пакет:** `internal/auth/`
**API:** `/api/v1/auth/*`, `/api/v1/mfa/*`

Outpost использует JWT-аутентификацию с поддержкой многофакторной аутентификации (MFA).

### JWT-токены

- Вход через `POST /api/v1/auth/login` возвращает JWT-токен
- Подпись: HMAC-SHA256 (`OUTPOST_JWT_SECRET`)
- TTL токена: 15 минут по умолчанию (`OUTPOST_TOKEN_TTL`)
- TTL сессии: 24 часа по умолчанию (`OUTPOST_SESSION_TTL`)
- Обновление: `POST /api/v1/auth/refresh`
- Rate limiting: 10 запросов/минуту на IP для auth-эндпоинтов

### Многофакторная аутентификация (MFA)

Поддерживаются три типа второго фактора:

| Тип | Описание | API |
|-----|----------|-----|
| **TOTP** | Временные одноразовые пароли (Google Authenticator, Authy) | `POST /mfa/totp/setup`, `POST /mfa/totp/verify` |
| **WebAuthn/FIDO2** | Аппаратные ключи безопасности (YubiKey, Touch ID) | `POST /mfa/webauthn/credentials` |
| **Резервные коды** | Одноразовые коды для аварийного доступа | `POST /mfa/backup-codes` |

Поток входа с MFA:
1. `POST /auth/login` -- возвращает `{"mfa_required": true, "session_token": "..."}`
2. `POST /auth/mfa/verify` -- отправляет код TOTP или WebAuthn-assertion с session_token
3. При успехе возвращается JWT-токен

### Управление сессиями

- `GET /api/v1/sessions` -- список активных сессий пользователя
- `DELETE /api/v1/sessions/{id}` -- завершение конкретной сессии
- `DELETE /api/v1/sessions` -- завершение всех сессий

### Внешние провайдеры аутентификации

| Провайдер | Описание |
|-----------|----------|
| **SAML 2.0** | Делегация аутентификации внешнему IdP (Okta, Azure AD, OneLogin) |
| **LDAP/AD** | Синхронизация пользователей и групп из Active Directory |
| **SCIM 2.0** | Автоматический провижининг пользователей из IdP |

## OIDC-провайдер

**Пакет:** `internal/auth/oidc/`

Outpost включает встроенный OpenID Connect провайдер идентификации. Это позволяет использовать Outpost как Identity Provider для сторонних приложений.

### Возможности

- Поддержка Authorization Code Flow с PKCE
- Выпуск ID-токенов и access-токенов
- Эндпоинт обнаружения: `/.well-known/openid-configuration`
- Регистрация OIDC-клиентов (relying parties)
- RSA-подпись токенов (ключ настраивается через `OUTPOST_OIDC_SIGNING_KEY`)

### Настройка

```bash
OUTPOST_OIDC_ISSUER=https://vpn.example.com
OUTPOST_OIDC_SIGNING_KEY=/path/to/rsa-private.pem  # опционально, генерируется при запуске
```

## Пользователи, группы и ACL

### Пользователи

**API:** `/api/v1/users/*`

- CRUD-операции: создание, просмотр, редактирование, удаление
- Активация/деактивация: `PATCH /users/{id}/activate`
- Поля: логин, email, имя/фамилия, роль, статус MFA
- Интеграция: LDAP/AD-синхронизация, SCIM-провижининг

### Группы

**API:** `/api/v1/groups/*`

- CRUD-операции над группами
- Управление членством: `POST/DELETE /groups/{id}/members`
- Предустановленные группы: `everyone`, `admins`

### ACL (Access Control Lists)

**API:** `/api/v1/groups/{id}/acls`

Контроль доступа на основе групп. ACL определяют, какие подсети доступны членам группы:

```json
{
  "group_id": "uuid",
  "network_id": "uuid",
  "allowed_ips": ["10.10.0.0/24", "192.168.1.0/24"],
  "description": "Доступ к офисной сети"
}
```

- `GET /groups/{id}/acls` -- список ACL группы
- `POST /groups/{id}/acls` -- создание ACL
- `DELETE /groups/{id}/acls/{aclId}` -- удаление ACL

## Сети, устройства и шлюзы

### Сети

**API:** `/api/v1/networks/*`

Сеть определяет WireGuard-пространство с CIDR, DNS и параметрами подключения:

```json
{
  "name": "office",
  "address": "10.10.0.0/16",
  "dns": ["1.1.1.1", "8.8.8.8"],
  "port": 51820,
  "keepalive": 25
}
```

### Устройства

**API:** `/api/v1/devices/*`

Устройство -- это WireGuard-пир, принадлежащий пользователю:

- Создание: `POST /devices` или `POST /devices/enroll`
- Одобрение/отзыв: `POST /devices/{id}/approve`, `POST /devices/{id}/revoke`
- Скачивание конфигурации: `GET /devices/{id}/config`
- Мои устройства: `GET /devices/my`

**Модель назначения:** Пользователь → Устройство → Сеть. Устройство привязывается к сети и получает IP из CIDR этой сети.

### Шлюзы

**API:** `/api/v1/gateways/*`

Шлюз -- это WireGuard-эндпоинт, обрабатывающий VPN-трафик:

- Создание с генерацией токена аутентификации
- Привязка к сети
- Мониторинг здоровья (last_seen)
- Подключение к core через gRPC с двунаправленным стримингом

## Zero Trust Network Access (ZTNA)

**Пакет:** `internal/ztna/`
**API:** `/api/v1/ztna/*`
**Фронтенд:** Виджет на дашборде + управление политиками

ZTNA реализует непрерывную верификацию устройств. Вместо предоставления постоянного сетевого доступа после начальной аутентификации, Outpost непрерывно оценивает состояние устройства и корректирует доступ соответственно.

### Проверки состояния устройств

Клиент Outpost отправляет данные о состоянии устройства на сервер core. Оцениваются следующие факторы:

| Фактор | Описание | Вес |
|--------|-------------|--------|
| Шифрование диска | FileVault (macOS), BitLocker (Windows), LUKS (Linux) | Высокий |
| Антивирус | Обнаружен активный антивирус/EDR | Средний |
| Версия ОС | Операционная система актуальна | Средний |
| Блокировка экрана | Блокировка экрана включена с таймаутом | Низкий |
| Файрвол | Хостовый файрвол активен | Средний |

### Оценка доверия

Каждое устройство получает оценку доверия (0-100), вычисляемую на основе результатов проверок:

```
Оценка доверия = sum(вес_фактора * прошёл_фактор) / sum(вес_фактора) * 100
```

Оценки доверия хранятся в таблице `trust_scores` и отслеживаются во времени (`trust_history`).

### Политики доступа

Политики ZTNA определяют минимальные оценки доверия для сетевого доступа:

```json
{
  "name": "require-encryption",
  "conditions": {
    "min_trust_score": 70,
    "require_disk_encryption": true
  },
  "action": "allow",
  "networks": ["uuid-чувствительной-сети"]
}
```

Когда оценка доверия устройства падает ниже порога политики, WireGuard-пир устройства удаляется со шлюза (применение на уровне протокола).

### DNS-правила

ZTNA включает DNS-фильтрацию на основе правил, которая может блокировать или перенаправлять DNS-запросы в зависимости от оценки доверия:

```
POST /api/v1/ztna/dns-rules
{
  "domain": "internal.example.com",
  "min_trust_score": 80,
  "action": "block"
}
```

### Конфигурация

Веса и пороги оценки доверия настраиваются через:
- **API:** `GET/PUT /api/v1/ztna/trust-config`
- **Панель администратора:** Настройки > Безопасность > ZTNA

## Site-to-Site туннели

**Пакет:** `internal/s2s/`
**API:** `/api/v1/s2s-tunnels/*`
**Фронтенд:** страница `/s2s`
**Подробная документация:** [mesh-networking.md](mesh-networking.md)

Соедините несколько офисных сетей через WireGuard-туннели с автоматическим обменом маршрутами.

### Топологии

#### Full Mesh

Каждый шлюз подключается напрямую к каждому другому шлюзу. Лучше всего для небольших сетей (до ~20 площадок).

```
  gw-A ◄────► gw-B
    ▲  \      /  ▲
    │    \  /    │
    │     X      │
    │   /  \     │
    ▼ /      \   ▼
  gw-C ◄────► gw-D
```

Количество пиров растёт как N*(N-1)/2.

#### Hub and Spoke

Все spoke-шлюзы подключаются к центральному хабу. Трафик между spoke-шлюзами маршрутизируется через хаб. Масштабируется до ~200 площадок.

```
        ┌─────┐
  ┌─────┤ Hub ├─────┐
  │     └──┬──┘     │
  │        │        │
  ▼        ▼        ▼
Spoke-A  Spoke-B  Spoke-C
```

### Изоляция интерфейсов

Каждый шлюз работает с двумя WireGuard-интерфейсами:
- `wg0` (порт 51820) -- клиентский VPN-трафик
- `wg1` (порт 51821) -- трафик S2S-туннелей

Это гарантирует, что компрометация S2S-ключа не повлияет на клиентские соединения, и к каждому интерфейсу могут быть применены независимые правила файрвола.

### Обмен маршрутами

Маршруты обмениваются через gRPC-стриминг между шлюзами и core:

1. Шлюз объявляет свои `local_subnets` core
2. Core вычисляет таблицу маршрутов на основе топологии (mesh: прямые маршруты; hub-spoke: маршруты через хаб)
3. Core отправляет вычисленные маршруты каждому шлюзу
4. Шлюз применяет маршруты к интерфейсу `wg1`

### Мониторинг здоровья

Шлюзы выполняют ICMP-проверки каждые 10 секунд:
- Ответ < 100 мс: **ЗДОРОВ**
- Ответ > 500 мс: **ДЕГРАДАЦИЯ**
- Нет ответа (3 раза): **НЕ РАБОТАЕТ**

Когда шлюз переходит в состояние НЕ РАБОТАЕТ, core пересчитывает маршруты, исключая отказавший шлюз, и отправляет обновления всем оставшимся шлюзам.

### Создание S2S-туннеля

Через панель администратора:
1. Перейдите в **S2S-туннели**
2. Нажмите **Новый туннель**
3. Выберите топологию (mesh или hub-spoke)
4. Добавьте шлюзы-участники с их локальными подсетями
5. Маршруты вычисляются и отправляются автоматически

Через API:
```bash
# Создание туннеля
curl -X POST http://localhost:8080/api/v1/s2s-tunnels \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"name": "office-mesh", "topology": "mesh"}'

# Добавление участников
curl -X POST http://localhost:8080/api/v1/s2s-tunnels/$TUNNEL_ID/members \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"gateway_id": "uuid", "local_subnets": ["10.1.0.0/24"]}'
```

## Smart Routes

**Пакет:** `internal/core/handler/smartroute.go`
**API:** `/api/v1/smart-routes/*`
**Фронтенд:** страница `/smart-routes`
**Таблицы БД:** `smart_routes`, `smart_route_entries`, `proxy_servers`, `network_smart_routes`

Smart Routes обеспечивают селективную прокси-маршрутизацию для обхода сетевых ограничений, аналогично dnsmasq + ipset.

### Группы маршрутов

Организуйте правила маршрутизации в группы, каждая из которых содержит записи для сопоставления трафика:

```json
{
  "name": "bypass-blocks",
  "description": "Маршрутизация заблокированных сервисов через прокси"
}
```

### Записи маршрутов

Каждая запись сопоставляет трафик по типу и применяет действие:

| Тип | Пример | Сопоставление |
|------|---------|---------|
| `domain` | `example.com` | Точное совпадение домена |
| `domain_suffix` | `.example.com` | Все поддомены example.com |
| `cidr` | `203.0.113.0/24` | Диапазон IP-адресов |

| Действие | Описание |
|--------|-------------|
| `proxy` | Маршрутизация через upstream прокси-сервер |
| `direct` | Прямое подключение (обход VPN для этого трафика) |
| `block` | Блокировка трафика |

### Прокси-серверы

Определите upstream прокси-серверы для действия `proxy`:

| Тип | Протокол |
|------|----------|
| `socks5` | SOCKS5-прокси |
| `http` | HTTP/HTTPS-прокси |
| `shadowsocks` | Шифрованный прокси Shadowsocks |
| `vless` | VLESS-прокси |

```json
{
  "name": "proxy-us",
  "type": "socks5",
  "address": "socks5://us-proxy.example.com:1080",
  "username": "user",
  "password": "pass"
}
```

### Привязка к сетям

Группы Smart Routes привязываются к конкретным сетям. Разные сети могут иметь разные правила маршрутизации:

```
POST /api/v1/smart-routes/{id}/networks/{networkId}
```

## Аналитика трафика

**Пакет:** `internal/analytics/`
**API:** `/api/v1/analytics/*`
**Фронтенд:** страница `/analytics`

Визуализация трафика в реальном времени и мониторинг пропускной способности.

### Эндпоинты

| Эндпоинт | Описание |
|----------|-------------|
| `GET /analytics/summary` | Общая пропускная способность, количество соединений, топ протоколов |
| `GET /analytics/bandwidth` | Временной ряд данных пропускной способности (поддерживает параметр интервала) |
| `GET /analytics/top-users` | Топ пользователей по потреблению трафика |
| `GET /analytics/connections-heatmap` | Паттерны соединений по времени суток |

### Источник данных

Данные аналитики собираются из таблицы `peer_stats`, партиционированной по `recorded_at` для эффективных запросов по временным диапазонам. Шлюзы отправляют статистику пиров в core через gRPC-стриминг, и core сохраняет их в этой таблице.

### График пропускной способности

Эндпоинт пропускной способности поддерживает настраиваемые интервалы:
```
GET /api/v1/analytics/bandwidth?from=2026-03-01T00:00:00Z&to=2026-03-15T00:00:00Z&interval=1h
```

Фронтенд отображает это как area chart (используя Recharts) на странице Аналитики и как мини-график на Дашборде.

### Тепловая карта

Тепловая карта соединений показывает, когда пользователи подключаются, помогая выявить паттерны использования и часы пиковой нагрузки. Отображается как сетка «день недели x час» на фронтенде.

## Панель соответствия стандартам

**Пакет:** `internal/compliance/`
**API:** `/api/v1/compliance/*`
**Фронтенд:** страница `/compliance`

Автоматическая оценка готовности к SOC 2, ISO 27001 и GDPR.

### Как это работает

Движок соответствия оценивает вашу конфигурацию Outpost по требованиям фреймворков и присваивает оценку готовности (0-100):

| Проверка | Фреймворки | Что проверяется |
|-------|-----------|------------------|
| Контроль доступа | SOC2, ISO27001 | RBAC настроен, количество администраторов ограничено |
| Принудительная MFA | SOC2, ISO27001, GDPR | MFA обязательна или доступна |
| Аудиторское логирование | SOC2, ISO27001 | Аудиторский журнал включён и настроено хранение |
| Шифрование данных в покое | SOC2, ISO27001, GDPR | SSL для базы данных включён |
| Шифрование данных при передаче | SOC2, ISO27001, GDPR | TLS настроен для API |
| Управление сессиями | SOC2 | TTL сессий настроен, выход работает |
| Хранение данных | GDPR | Политики хранения данных определены |
| Согласие пользователей | GDPR | Механизмы согласия реализованы |
| Ротация ключей | ISO27001 | Ротация ключей PKI настроена |
| Сегментация сети | SOC2, ISO27001 | Сетевые ACL настроены |

### Эндпоинты

```bash
# Полный отчёт (все фреймворки)
GET /api/v1/compliance/report

# По отдельным фреймворкам
GET /api/v1/compliance/soc2
GET /api/v1/compliance/iso27001
GET /api/v1/compliance/gdpr
```

### Формат ответа

```json
{
  "framework": "soc2",
  "score": 78,
  "max_score": 100,
  "checks": [
    {
      "id": "soc2-access-control",
      "name": "Контроль доступа",
      "status": "pass",
      "details": "RBAC включён, 2 администраторских аккаунта"
    },
    {
      "id": "soc2-mfa",
      "name": "Многофакторная аутентификация",
      "status": "warn",
      "details": "MFA доступна, но не обязательна"
    }
  ]
}
```

Значения статуса: `pass`, `warn`, `fail`

## Мультитенантность / MSP-режим

**Пакет:** `internal/tenant/`
**API:** `/api/v1/tenants/*`
**Фронтенд:** управление тенантами на уровне администратора

Запускайте один экземпляр Outpost, обслуживающий несколько организаций с полной изоляцией данных. Спроектировано для Managed Service Providers (MSP) и реселлеров.

### Управление тенантами

```bash
# Создание тенанта
POST /api/v1/tenants
{
  "name": "Acme Corp",
  "slug": "acme",
  "admin_email": "admin@acme.com"
}

# Список тенантов
GET /api/v1/tenants

# Статистика тенанта
GET /api/v1/tenants/{id}/stats
```

### Изоляция данных

У каждого тенанта изолированы:
- Пользователи и группы
- Сети и устройства
- Шлюзы и S2S-туннели
- Аудиторские логи

## Встроенная PKI

**Пакет:** `internal/pki/`

Автоматическая ротация ключей WireGuard без простоев.

### Как это работает

1. Движок PKI генерирует новые пары ключей WireGuard по настраиваемому расписанию
2. Новые ключи распространяются на шлюзы через gRPC-стриминг
3. Устройства получают новые конфигурации через API
4. Старые ключи выводятся из эксплуатации после grace-периода
5. Ротация без простоев: старые и новые ключи сосуществуют во время перехода

### Поток ротации ключей

```
Расписание запускает ротацию
        │
        ▼
PKI генерирует новую пару ключей для шлюза
        │
        ▼
Core отправляет новый ключ на шлюз через gRPC
        │
        ▼
Шлюз настраивает новый ключ рядом со старым
        │
        ▼
Устройства получают обновлённую конфигурацию (новый публичный ключ пира)
        │
        ▼
После grace-периода старый ключ удаляется
```

### Преимущества

- Ограничивает последствия компрометации ключа
- Автоматизировано -- ручное вмешательство не требуется
- Без простоев при ротации
- Полный аудиторский журнал событий жизненного цикла ключей

## MFA на уровне протокола WireGuard

В отличие от MFA на уровне приложения (где VPN-туннель остаётся активным после истечения сессии), Outpost реализует MFA на уровне протокола WireGuard.

### Как это работает

Когда сессия пользователя истекает или требуется повторная верификация MFA:

1. Core обнаруживает истечение сессии или падение оценки доверия
2. Core отправляет `PeerUpdate` (action=REMOVE) на все шлюзы через StreamHub
3. Шлюз удаляет пир пользователя из WireGuard-интерфейса
4. VPN-туннель немедленно разрывается на уровне протокола
5. Пользователь должен пройти повторную аутентификацию с MFA для переподключения
6. После успешной повторной аутентификации пир возвращается на шлюзы

### Сравнение

| Аспект | MFA на уровне приложения | MFA на уровне протокола (Outpost) |
|--------|----------------------|------------------------------|
| Туннель после истечения сессии | Остаётся активным | Разрывается |
| Точка применения | Прикладной уровень | WireGuard-интерфейс |
| Возможность обхода | Потенциально да | Нет -- пир удалён со шлюза |
| Повторная аутентификация | При следующем API-вызове | Необходимо восстановить туннель |

### Конфигурация

MFA на уровне протокола работает автоматически при включённой MFA для пользователя. TTL сессии (`OUTPOST_SESSION_TTL`, по умолчанию 24 часа) определяет, как часто требуется повторная аутентификация.

Для принудительной MFA для всех пользователей:
1. Перейдите в **Настройки** > **Безопасность**
2. Включите **Требовать MFA**
3. Все пользователи должны будут настроить TOTP или WebAuthn при следующем входе

## Визуальная карта сети

**Фронтенд:** страница Дашборда (`/`)
**Источники данных:** API шлюзов, API устройств, API S2S-туннелей

Дашборд включает интерактивную визуализацию сетевой топологии, показывающую всю VPN-сеть в реальном времени.

### Элементы

| Элемент | Форма | Цвет | Представляет |
|---------|-------|-------|------------|
| Core | Ромб | Синий | Центральный control plane |
| Шлюз | Квадрат | Зелёный (онлайн) / Красный (офлайн) | WireGuard-эндпоинты |
| Устройство | Круг | Зелёный (подключено) / Серый (отключено) | Клиентские пиры |
| S2S-связь | Пунктирная линия | Синий | Активный S2S-туннель |
| gRPC-связь | Сплошная линия | Серый | Соединение control plane |

### Обновление данных

Карта сети получает данные из:
- `GET /api/v1/gateways` -- список шлюзов и статус здоровья
- `GET /api/v1/devices` -- список устройств и статус подключения
- `GET /api/v1/s2s-tunnels` -- активные туннели и участники

Фронтенд периодически опрашивает эти эндпоинты (через TanStack Query) для поддержания актуальности визуализации.

### Взаимодействие

- Нажмите на шлюз, чтобы увидеть его детали (подключённые пиры, пропускная способность, здоровье)
- Нажмите на устройство, чтобы увидеть его детали (пользователь, IP, последний хендшейк)
- Наведите курсор на S2S-связь, чтобы увидеть имя туннеля и статус здоровья
- Масштабирование и перемещение для навигации по большим топологиям

## Аудиторское логирование

**Пакет:** `internal/observability/`
**API:** `/api/v1/audit/*`
**Таблица БД:** `audit_log`

Все изменяющие HTTP-запросы (POST, PUT, PATCH, DELETE) автоматически логируются через `observability.AuditMiddleware`.

### Данные аудиторской записи

Каждая запись содержит:
- Временная метка
- ID пользователя, выполнившего действие
- Действие (create, update, delete)
- Тип ресурса и ID ресурса
- Детали запроса (JSONB)
- IP-адрес клиента и User-Agent

### Эндпоинты

| Эндпоинт | Описание |
|----------|----------|
| `GET /audit` | Список записей с фильтрацией (по пользователю, действию, ресурсу, дате) |
| `GET /audit/stats` | Статистика аудита (количество по типам действий) |
| `GET /audit/export` | Экспорт записей аудита |

## Вебхуки

**Пакет:** `internal/webhook/`
**API:** `/api/v1/webhooks/*`

Вебхуки позволяют отправлять HTTP-уведомления о событиях на внешние системы (Slack, PagerDuty, SIEM и т.д.).

### Управление

- `GET /webhooks` -- список вебхуков
- `POST /webhooks` -- создание (URL, события, секрет для подписи)
- `GET /webhooks/{id}` -- детали вебхука
- `DELETE /webhooks/{id}` -- удаление
- `POST /webhooks/{id}/test` -- отправка тестового события

### События

Вебхуки могут подписываться на события: создание/удаление пользователей, одобрение/отзыв устройств, изменения сетей, сбои аутентификации и другие.

## NAT Traversal

**Пакет:** `internal/nat/`
**API:** `/api/v1/nat/*`

NAT traversal помогает установить WireGuard-соединения через NAT и файрволы.

### Возможности

- Определение типа NAT: `POST /nat/check`
- Управление relay-серверами (STUN/TURN): `GET/POST /nat/relays`, `DELETE /nat/relays/{id}`
- Статус NAT: `GET /nat/status`

### Конфигурация

```bash
OUTPOST_NAT_ENABLED=true
OUTPOST_STUN_PORT=3478
OUTPOST_TURN_PORT=3479
OUTPOST_TURN_REALM=outpost
OUTPOST_EXTERNAL_IP=  # авто-определение, если пусто
```

## Наблюдаемость

**Пакет:** `internal/observability/`

### Метрики Prometheus

Доступны по адресу `/metrics` (без аутентификации).

| Метрика | Тип | Описание |
|---------|-----|----------|
| `outpost_active_peers` | Gauge | Текущие подключённые WireGuard-пиры |
| `outpost_bandwidth_bytes_total` | Counter | Общий трафик (метки rx/tx) |
| `outpost_gateway_last_seen_seconds` | Gauge | Время с последнего heartbeat шлюза |
| `outpost_auth_attempts_total` | Counter | Попытки аутентификации (метки success/failure) |
| `outpost_s2s_tunnel_status` | Gauge | Состояние S2S-туннеля (1=работает, 0=не работает) |

### Интеграция с SIEM

Outpost поддерживает отправку событий безопасности во внешние SIEM-системы через вебхуки. Аудиторские логи, события аутентификации и изменения конфигурации могут быть перенаправлены в Splunk, Elastic SIEM, Sumo Logic и другие.

### Структурированное логирование

Все компоненты используют `slog` с настраиваемым уровнем и форматом:
- **Уровни:** `debug`, `info`, `warn`, `error`
- **Форматы:** `json` (продакшен), `text` (разработка)
- **Настройка:** `OUTPOST_LOG_LEVEL`, `OUTPOST_LOG_FORMAT`
